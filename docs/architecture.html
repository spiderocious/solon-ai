Start here

This document is the canonical map of how Solon — a political intelligence & campaign platform for Nigeria — is going to be built. It pairs the product spec in docs/product/mvp.md with concrete engineering choices: components, data flow, what we keep, what we trim, and why.

The big picture

Solon is a modular monolith with a handful of background workers and a public edge. Five product modules share one backend, one database, one job queue, one object store. We do not split services until a real reason forces the split.

Why a modular monolith? Microservices add latency, deploy complexity, and a cross-service type-safety tax for zero benefit at MVP scale. Five product modules sharing one Postgres with proper schemas costs you nothing today and lets you split later exactly where the seams actually appear. The cost of premature splitting (today's data-layer) is real; the cost of deferring the split is near zero if you keep clean module boundaries.

Design principles

1. Boring tech, audit-first

Postgres, Express, Redis, an object store. Every write is an append to an audit_log row. NDPR compliance is a feature.

2. Module per directory, one DB

Each MVP module lives at src/modules/<name>. Schemas are namespaced (sim.*, voter.*, fin.*). Splitting later means moving a directory + a schema.

3. Idempotency on every write

2G/3G + offline queueing means duplicate submissions are normal. Every state-changing endpoint accepts Idempotency-Key and dedupes server-side.

4. Workers do the slow stuff

OCR, STT, scenario simulation, donor screening, weekly digests — all queue jobs. The HTTP path stays under 200ms.

5. Buy ML, build the model

Whisper for STT. GPT-4o vision for EC8A OCR. Termii for SMS. ComplyAdvantage for PEP. The political model in Module 0 is the only ML we own.

6. Citizen path is rate-limited at the edge

Public Citizen Reporter traffic terminates at Cloudflare with rate limits + trust scoring before reaching origin. Origin never sees abusive traffic.

System topology

The runtime split is intentionally minimal. Three deployable units: the API, the workers, and the edge.

Why one API process
Five product modules at MVP scale fit comfortably in one Node process. Module boundaries are enforced by directory structure and DB schemas, not network calls. If the Simulator's load profile ever diverges hard from War Room's, you split then — moving a folder and rotating a secret.
Why a separate workers process
OCR, STT, and simulation runs take seconds-to-minutes. Running them in the HTTP process would exhaust the event loop, blow your p99, and crash on a single bad image. A separate process with its own scaling lever (Render worker service) is non-negotiable.
Why one Postgres
Cross-module joins are real: finance needs to know which campaign, which constituency; war room needs polling-unit history from voter-data; the simulator needs every cycle's results. Splitting DBs means rebuilding those joins as HTTP calls. PostGIS handles geo, pgvector handles embeddings, JSONB handles AI extraction blobs — all in one engine.
Why Redis
BullMQ (queue) + simple cache + pub/sub for SSE fan-out, all in one box. Render hosts it. We are not introducing Kafka/NATS for an MVP — Redis pub/sub is enough for the war-room subscriber count we'll see in 2027 cycles.
Why R2 / object store
EC8A images can be megabytes; one polling unit can generate dozens of submissions; receipts pile up. We never put binary blobs in Postgres. R2's free egress beats S3 hard for a Nigeria-served app — though if you'd rather keep everything on Render, Render's persistent disks + an S3-compatible add-on works for v1.
Why an edge tier
Citizen Reporter is publicly accessible (mvp.md §4 line 247). That's an abuse magnet. Cloudflare's free tier gives you DDoS protection, rate limiting per IP/phone, and the option of running cheap trust-scoring at the edge before traffic reaches Render. Render alone doesn't have this. If we drop Cloudflare we accept higher exposure to junk traffic on election day.

202-first pattern: HTTP returns fast, workers do the slow lift, SSE pushes the result back.
Why 202-first? Agent on a 2G connection cannot wait 8 seconds for an LLM call. We acknowledge the submission, queue the work, push the result over SSE when the worker finishes. This same pattern is reused for every module: simulator runs, donor screening, voice note transcription, weekly digests.

Why namespaced schemas instead of separate DBs? Same physical DB, but the schema boundary forces every cross-module read to be deliberate. If you later split a service out, the migration is mechanical — copy the schema. No global table soup.
Tenancy: campaigns never see each other's data

Every campaign-scoped row carries a campaign_id. A request-scoped guard reads the user's membership and refuses any query without a matching campaign_id filter. Implementation: a per-request Postgres SET LOCAL app.current_campaign = ... and a small repo helper that injects the filter into every query.

We are not using Postgres Row-Level Security in v1. Why: operational cost (every connection pooler config, every migration, every test) outweighs the marginal safety over app-layer enforcement when you have a single API process. Revisit at v2 if a second service joins the trust zone.

Why JWT + refresh, not sessions

The agent and citizen flows are mobile-first on flaky networks. A bearer token in localStorage (web) or secure storage (PWA) survives reconnects better than a cookie session. Refresh tokens rotate. Access tokens are short (15m) and contain only {user_id, role, memberships[]}; the authz layer re-checks on every request against the membership table — so revoking access doesn't require token blacklists.

OTP throttling — the abuse vector you'll forget

SMS costs money and OTP endpoints are the #1 way bad actors burn your runway. Hard caps in iam.otp_attempts: 3 attempts per phone per 15 minutes, 10 per phone per day, 200 per IP per hour. Cloudflare adds a global rate limit on top. Every blocked attempt is logged to audit.events.

Offline & sync (Modules 2 + 4)

2G/3G submission queueing is a hard MVP requirement (mvp.md line 280). The protocol:

+ generate idempotency_key (uuid) Note over SW: Network returns SW->>API: POST /submissions w/ Idempotency-Key header alt First time seeing key API->>PG: INSERT submission(idempotency_key, ...) ON CONFLICT DO NOTHING PG-->>API: inserted API-->>SW: 202 { id, status: pending } else Key already exists API->>PG: SELECT by idempotency_key PG-->>API: existing row API-->>SW: 202 { id, status: } // idempotent replay end SW->>App: mark queued submission as synced
Idempotent submission protocol — duplicates from retries collapse server-side.
Why client-generated idempotency keys? The client has the only stable identity for a submission across retries. If the server generated IDs, a retried request would create duplicates. The client generates a UUID v4 once when the submission is captured, stores it in IndexedDB next to the payload, and sends it on every retry. The DB has a unique index on (idempotency_key, campaign_id).

Real-time & background jobs

The war room (Module 4) and the simulator (Module 0) need different real-time models.

War Room: SSE + Redis pub/sub

Every CM connected to GET /war-room/:id/stream opens a Server-Sent Events connection. Workers publish updates to a Redis channel. The API subscribes and fans out. Why SSE not WebSocket: one-way push is enough for war-room data; SSE auto-reconnects; works through every proxy.

Simulator: BullMQ + polling

Scenario runs take 1–10s. Submit returns 202 { runId }. Client polls GET /runs/:id every 1s, or — if already on a war-room SSE channel — receives push. Why a queue at all: isolates the model from HTTP, lets us run scenarios on different worker instances later.

Weekly digests & anomalies

BullMQ repeatable jobs. Issue salience weekly. Donor anomaly weekly. PEP screening on-write + nightly re-screen. All log to audit.events.

Audit & compliance (NDPR)

mvp.md §cross-cutting line 282 mandates audit of every data access and admin action. This is not optional and not retrofittable.

How we log every meaningful event

Two layers:

  1. Application-level: a thin audit() helper called from every mutating service method. Writes one row to audit.events in the same transaction as the business write. If the audit insert fails, the business write rolls back. This is the only way to guarantee no "silent" mutation.
  2. DB-level safety net: Postgres trigger on sensitive tables (iam.users, fin.donations, fin.expenses, wr.submissions) writes a before/after JSONB delta to audit.events. Belt-and-braces — catches anything the application layer forgets.

Reads of voter data also log: every SELECT against voter.* goes through a repo helper that records (user_id, query_shape, row_count, requested_at). Performance cost is one append per query — acceptable.

Why both layers? The app layer gives you semantic context ("user X approved donation Y"). The DB trigger gives you tamper-proofing — you cannot mutate a row through psql without leaving a trace. NDPR auditors will ask for the latter.
Data export & deletion (line 283)

Per-campaign export: a worker assembles every row touching that campaign_id into a signed ZIP in R2, emails the campaign owner a 7-day link. Deletion: 30-day grace window after request; rows are soft-deleted with deleted_at, then hard-purged by a nightly worker, except audit.events which is retained for the statutory minimum (default 7 years).

What "simulation engine" actually is

For v1 keep it deliberately simple — a ridge regression / gradient-boosted model trained on 2015/2019/2023 INEC results + demographic data per polling unit, predicting per-party vote share. Levers (turnout, defection, withdrawal) apply as multiplicative adjustments with learned coefficients. Issue shocks apply as additive deltas with a decay function.

This lives in a Python workers/sim sub-process called by BullMQ, not in Node. Reason: pandas/scikit/numpy are non-negotiable for this kind of model and rewriting them in TS is foolish. The Node worker shells out to Python via a tiny HTTP server on localhost or a child-process IPC.

Why not a fancier model? Because the value is in explainability: top-5 drivers per projection, written rationale, confidence band. A simple model whose features you can cite beats a deep net you can't. v2 can add ensembles and Bayesian intervals — v1 needs to be defensible in print.
The NL Copilot — RAG over your own model, not the web

The copilot doesn't answer freeform political questions. It does two things:

  1. Parse: map natural-language scenarios to structured inputs ({candidates, levers, shocks}) using a tool-calling LLM. Failures fall back to "I couldn't parse that — please use the form".
  2. Explain: given a run's outputs + top features, generate a one-paragraph rationale. Inputs to the LLM are the model's actual numbers, not the user's prose — grounded explanation, no hallucinated facts.

"Why did LP get a 6-point bump?" routes to the second prompt with the run's feature importances attached. The LLM has no internet access, no other data — just the run's JSON.

Ethical guardrails — what we actually block

A small classifier (regex + zero-shot LLM check) on every scenario input. Blocks:

  • Negative turnout adjustments targeted at named demographics in opposition strongholds.
  • "How do I depress …" / "how do I suppress …" NL prompts.
  • Scenarios that request individual voter targeting (we don't have the data anyway, but we log the attempt).

Every block writes to audit.events with the raw prompt. Repeat offenders get the account flagged for admin review.

The election-day load curve

Nigeria has ~176k polling units. Realistic MVP target: support a single Senate / gubernatorial race with ~2k-5k polling units of interest. Even at 10x submission redundancy that's 20k-50k inbound events in a 4-6 hour window — roughly 2-3 writes/sec sustained, with bursts of 20-50/sec.

One Render Standard instance + Postgres + a 2-worker pool handles this easily. The constraint is not throughput, it's the LLM rate limit on OCR — solve via batch queueing and a vendor with high concurrent limits (OpenAI Tier 4, Anthropic prod tier).

What about object storage on Render?

Render doesn't offer S3-compatible object storage. Two options:

  1. Cloudflare R2 (recommended) — free egress, S3-compatible API, $0.015/GB-month storage. Set up presigned uploads from the API; agents/citizens upload direct to R2, never proxy through your API. Saves bandwidth and CPU.
  2. Backblaze B2 — also S3-compatible, slightly cheaper storage, ~$0.01/GB egress (vs R2's free). Fine fallback.

Do not use Render persistent disks for user uploads. They don't autoscale and the pricing model is bad.

CI/CD shape

GitHub Actions → on push to main, Render auto-deploys the service whose path was touched (Render supports per-service path filters). Migrations run on deploy via a pre-deploy script. Add a single end-to-end smoke test that hits /healthz + one simulator run on every deploy.
